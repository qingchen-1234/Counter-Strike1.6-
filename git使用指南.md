好的，我们开始！下面按你实际写代码、用 GitHub 管理代码的**真实流程**，把命令分成六大模块，一条一条讲透。

## 模块一：环境配置与本地仓库初始化

**这个模块解决什么问题？**
就是让你的电脑知道“你是谁”，并让一个普通文件夹变成“受 Git 管理的项目文件夹”（本地仓库）。相当于先领工牌，再把工位收拾好，准备干活。

---

### 【环境配置】命令 1：`git config --global user.name "你的名字"`
#### 1. 总体解释
告诉 Git 你叫什么名字。之后你每一次提交代码，都会附上这个名字，方便队友知道是谁改的。**装好 Git 之后第一件事就是做这个，只需做一次。**

#### 2. 命令逐词/逐部分拆解
- **`git`**：调用 Git 这个版本控制工具。
- **`config`**：修改 Git 的配置。
- **`--global`**：表示这个配置对你电脑上的所有 Git 项目都有效（全局生效），而不是只对当前项目。
- **`user.name`**：配置项的名字，这里代表“用户名”。
- **`"你的名字"`**：你想让 Git 记住的名字，建议用英文或拼音，比如 `"Xiao Ming"`。

#### 3. 完整使用示例
```bash
git config --global user.name "Xiao Ming"
```

#### 4. 常接/前置关联命令
- **之后**：通常紧接着配置邮箱，就是下面这条命令。

---

### 【环境配置】命令 2：`git config --global user.email "你的邮箱"`
#### 1. 总体解释
告诉 Git 你的邮箱。和上面的名字配套使用，也是全局配置一次就行。

#### 2. 命令逐词/逐部分拆解
- **`git`**：调用 Git。
- **`config`**：修改配置。
- **`--global`**：全局生效。
- **`user.email`**：配置项叫“用户邮箱”。
- **`"你的邮箱"`**：填写你注册 GitHub 用的那个邮箱。

#### 3. 完整使用示例
```bash
git config --global user.email "xiaoming@example.com"
```

#### 4. 常接/前置关联命令
- **之后**：可以用 `git config --list` 检查配置是否生效。

---

### 【环境配置】命令 3：`git config --list`
#### 1. 总体解释
查看你当前所有的 Git 配置，像查工牌信息一样，看看名字、邮箱有没有设对。

#### 2. 命令逐词/逐部分拆解
- **`git config`**：操作配置。
- **`--list`**：列出所有配置项和它的值。

#### 3. 完整使用示例
```bash
git config --list
```

#### 4. 常接/前置关联命令
- **之前**：通常配置完名字、邮箱后用来检查一下。

---

### 【本地仓库初始化】命令 4：`git init`
#### 1. 总体解释
把当前这个普通文件夹，变成一个 Git 本地仓库。**将来你的所有版本记录都会存在这个文件夹里一个叫 `.git` 的隐藏目录里。** 每次新建项目时，只需在这个项目文件夹里执行一次。

#### 2. 命令逐词/逐部分拆解
- **`git`**：调用 Git。
- **`init`**：是 initialize（初始化）的缩写，意思是“在这里生成一个全新的 Git 仓库”。

#### 3. 完整使用示例
```bash
# 先进入你的项目文件夹
cd my-project
# 然后把这个文件夹初始化成 Git 仓库
git init
```

#### 4. 常接/前置关联命令
- **之前**：必须先用 `cd` 进入你想管理的项目文件夹。
- **之后**：通常会开始写代码，然后用 `git add` 把文件加入暂存区。

---

## 模块二：远程仓库关联与绑定

**这个模块解决什么问题？**
你电脑上的本地仓库，和 GitHub 网站上的远程仓库本来是两回事。你需要把它们“牵线搭桥”，从这以后才能把本地代码推送到 GitHub，或者从 GitHub 拉取别人更新的代码。

---

### 【远程仓库关联】命令 5：`git remote add origin 仓库地址`
#### 1. 总体解释
给你的本地仓库添加一个“远程仓库链接”，并给这个链接起个外号叫 `origin`（习惯上叫这个），以后你推送或拉取代码时，就直接用 `origin` 代替那一长串网址。

#### 2. 命令逐词/逐部分拆解
- **`git remote`**：管理远程仓库连接。
- **`add`**：添加一个远程仓库链接。
- **`origin`**：给这个远程仓库链接起的**别名**（可以理解为外号）。习惯上大家都用 `origin`，你也可以换成别的名字，但别换，省得出错。
- **`仓库地址`**：GitHub 上你仓库的网址，通常以 `https://github.com/` 开头，比如 `https://github.com/你的用户名/项目名.git`。

#### 3. 完整使用示例
```bash
git remote add origin https://github.com/xiaoming/my-project.git
```

#### 4. 常接/前置关联命令
- **之前**：你得先在 GitHub 网站上创建一个空仓库（不要勾选“添加 README”），复制它的地址。本地文件夹也得先用 `git init` 初始化好。
- **之后**：可以用 `git remote -v` 检查是否绑定成功。

---

### 【远程仓库关联】命令 6：`git remote -v`
#### 1. 总体解释
查看当前本地仓库绑定了哪些远程仓库，以及它们的完整地址。`-v` 是“详细模式”，会把别名和对应的拉取、推送地址都列出来。

#### 2. 命令逐词/逐部分拆解
- **`git remote`**：管理远程仓库。
- **`-v`**：是 `--verbose`（啰嗦模式）的缩写，让 Git 显示更多信息。

#### 3. 完整使用示例
```bash
git remote -v
```
正常会显示类似：
```
origin  https://github.com/xiaoming/my-project.git (fetch)
origin  https://github.com/xiaoming/my-project.git (push)
```

#### 4. 常接/前置关联命令
- **之前**：刚用 `git remote add` 绑定了远程地址。
- **之后**：如果地址错了，可以用 `git remote set-url` 修改。

---

## 模块三：代码上传（本地 → 远程）完整流程

**这个模块解决什么问题？**
把你本地写好的代码，安全地传到 GitHub 远程仓库上保存。这是平时开发**最常用的一套组合拳**。

---

### 【代码上传流程第一步】命令 7：`git status`
#### 1. 总体解释
看一看当前工作区里有什么变动：哪些文件是新加的、哪些被修改了、哪些正准备提交。这是一个**纯查看命令，不会修改任何东西**，随时可用。

#### 2. 命令逐词/逐部分拆解
- **`git`**：调用 Git。
- **`status`**：状态，意思是“给我汇报一下当前情况”。

#### 3. 完整使用示例
```bash
git status
```
它会告诉你，比如“你有一个新文件 `index.html` 还没被跟踪”。

#### 4. 常接/前置关联命令
- **之前**：改完代码后，准备提交前看一眼，养好习惯。
- **之后**：根据提示，决定用 `git add` 把哪些文件加入暂存区。

---

### 【代码上传流程第二步】命令 8：`git add 文件名` 或 `git add .`
#### 1. 总体解释
把你指定的文件，从“工作区”（就是你的文件夹）放入“暂存区”（可以理解为一个“准备提交的购物车”）。只有放进暂存区的文件，下次提交才会被记录。

#### 2. 命令逐词/逐部分拆解
- **`git add`**：添加文件到暂存区。
- **`文件名`**：比如 `index.html`，就是只添加这一个文件。
- **`.`**：一个英文句号，代表**当前文件夹下所有变动的文件**。用 `git add .` 就是“把刚才改过的所有文件全加进购物车”。

#### 3. 完整使用示例
```bash
# 只添加一个文件
git add index.html

# 添加所有改动的文件（最常用）
git add .
```

#### 4. 常接/前置关联命令
- **之前**：通常先用 `git status` 看看改了啥，再决定用 `add` 加哪些。
- **之后**：接着就要用 `git commit` 来提交这个“购物车”里的东西。

---

### 【代码上传流程第三步】命令 9：`git commit -m "提交说明"`
#### 1. 总体解释
把暂存区（购物车）里的所有文件正式提交到本地仓库，生成一个版本记录，并附上一条说明标签，告诉你和队友这次改了什么。

#### 2. 命令逐词/逐部分拆解
- **`git commit`**：提交，正式记录一个版本。
- **`-m`**：是 `--message` 的缩写，后面接一个字符串作为提交说明。
- **`"提交说明"`**：双引号里的文字就是你对这次修改的描述。**一定写清楚你为什么改，不要只写“修改”**，比如 `"修复了登录按钮无法点击的bug"`。

#### 3. 完整使用示例
```bash
git commit -m "完成了首页导航栏的样式编写"
```

#### 4. 常接/前置关联命令
- **之前**：必须先用 `git add` 把要提交的文件加入暂存区。
- **之后**：
  - 如果是第一次推送这个分支，接着用 `git push -u origin main`。
  - 如果之前已经推送过，直接用 `git push`。

---

### 【代码上传流程第四步】命令 10：`git push -u origin main`
#### 1. 总体解释
把你本地仓库的提交记录，推送到 GitHub 远程仓库的 `main` 分支上。**第一次推送这个分支时**用 `-u` 参数，可以把本地的 `main` 分支和远程的 `main` 分支绑定起来，以后只需敲 `git push` 就行。

#### 2. 命令逐词/逐部分拆解
- **`git push`**：推送，上传本地版本到远程。
- **`-u`**：是 `--set-upstream` 的缩写，意思是“设置上游分支并记住这个绑定关系”。
- **`origin`**：远程仓库的别名，就是之前我们绑定的那个。
- **`main`**：你要推送到远程仓库的哪个分支。现在 GitHub 默认主分支叫 `main`（以前叫 `master`）。

#### 3. 完整使用示例
```bash
git push -u origin main
```
第一次执行后，Git 就记住了本地 `main` 对应远程 `origin/main`。之后在 `main` 分支上再推送，只需：
```bash
git push
```

#### 4. 常接/前置关联命令
- **之前**：本地必须已经有 `git commit` 产生的提交记录。
- **之后**：可以去 GitHub 网页上刷新一下，看到代码已经上去了。

---

### 【代码上传补充】命令 11：`git push`（已绑定分支的常规推送）
#### 1. 总体解释
把当前分支的本地新提交，直接推送到它绑定的远程分支上。**前提是你之前已经用 `git push -u` 绑定过了。**

#### 2. 命令逐词/逐部分拆解
- **`git push`**：推送。

#### 3. 完整使用示例
```bash
git push
```

#### 4. 常接/前置关联命令
- **之前**：必须已经绑定过上游分支，或者你在 `git push` 后面手动指定远程和分支名。

---

## 模块四：分支管理（创建、切换、合并、删除）

**这个模块解决什么问题？**
让你可以平行开发多个功能而不互相干扰。比如你可以从主分支劈出一条“开发支付功能”的分支，支付开发完再合并回主分支。

---

### 【分支管理-查看分支】命令 12：`git branch`
#### 1. 总体解释
查看本地所有分支。当前所在的分支前面会有一个 `*` 号并高亮。

#### 2. 命令逐词/逐部分拆解
- **`git branch`**：列出所有本地分支。

#### 3. 完整使用示例
```bash
git branch
```
输出示例：
```
* main
  feature-login
```

#### 4. 常接/前置关联命令
- **之后**：知道自己在哪个分支后，可能会用 `git checkout 分支名` 切换分支。

---

### 【分支管理-创建分支】命令 13：`git branch 新分支名`
#### 1. 总体解释
基于当前所在分支，**创建**一个新的分支，但不会自动切换过去。相当于你照着当前分支的样子，复制一条新的时间线出来。

#### 2. 命令逐词/逐部分拆解
- **`git branch`**：操作分支。
- **`新分支名`**：你想创建的分支的名字，建议用英文表示用途，比如 `feature-payment`。

#### 3. 完整使用示例
```bash
git branch feature-payment
```

#### 4. 常接/前置关联命令
- **之后**：需要用 `git checkout feature-payment` 或 `git switch feature-payment` 切换到这个新分支。

---

### 【分支管理-切换分支】命令 14：`git checkout 分支名` （或 `git switch 分支名`）
#### 1. 总体解释
从当前分支，**切换**到另一个分支。切换后你的文件夹里的代码会变成那个分支的样子。

#### 2. 命令逐词/逐部分拆解
- **`git checkout`**：检出，就是切换到某个分支或某个历史版本。
- **`分支名`**：目标分支的名字。
> 新版 Git 也推荐用 `git switch 分支名` 专门做切换，更安全，意思相同。

#### 3. 完整使用示例
```bash
git checkout feature-payment
# 或者
git switch feature-payment
```

#### 4. 常接/前置关联命令
- **之前**：务必确保当前分支的改动已经 `commit` 或 `stash`（暂存），否则可能带过去造成混乱。
- **之后**：开始在新分支上写代码，然后 `add`，`commit`。

---

### 【分支管理-创建并切换】命令 15：`git checkout -b 新分支名`（或 `git switch -c 新分支名`）
#### 1. 总体解释
一步到位：**创建新分支，并立刻切换过去**。是把 `git branch` 和 `git checkout` 合并的一个快捷方式。

#### 2. 命令逐词/逐部分拆解
- **`git checkout -b`**：`-b` 表示“创建并切换分支”。
- **`新分支名`**：你新建分支的名字。
- 同理 `git switch -c`：`-c` 表示 create 并切换。

#### 3. 完整使用示例
```bash
git checkout -b feature-payment
# 或者
git switch -c feature-payment
```
执行后，你会直接处于新分支上。

#### 4. 常接/前置关联命令
- **之前**：通常在主分支（如 `main`）上，确保主分支是最新代码。
- **之后**：直接开始写代码。

---

### 【分支管理-合并分支】命令 16：`git merge 被合并的分支名`
#### 1. 总体解释
把另一个分支上的代码改动，合并到**你当前所在的分支**上。比如你在 `main` 分支，想把 `feature-payment` 上的代码加进来，就用这个命令。

#### 2. 命令逐词/逐部分拆解
- **`git merge`**：合并分支。
- **`被合并的分支名`**：你想把哪个分支的改动拿过来，比如 `feature-payment`。

#### 3. 完整使用示例
```bash
# 先切换到要“接收改动”的分支，比如 main
git checkout main
# 把 feature-payment 的改动合并进来
git merge feature-payment
```

#### 4. 常接/前置关联命令
- **之前**：一定要先切换到目标分支（接收方）。建议合并前都 `git status` 确认工作区干净。
- **之后**：如果没有冲突，直接完成；如果有冲突，需要手动解决冲突后，再 `add` 和 `commit`。

---

### 【分支管理-删除分支】命令 17：`git branch -d 分支名`
#### 1. 总体解释
删除一个**已经合并过、不需要了**的本地分支，保持分支列表干净。

#### 2. 命令逐词/逐部分拆解
- **`git branch -d`**：`-d` 是 `--delete` 的缩写，安全删除，如果你没合并它会提醒你。
- **`分支名`**：要删除的分支名字。

#### 3. 完整使用示例
```bash
git branch -d feature-payment
```
如果分支还没合并但硬要删，用 `-D`（大写）。

#### 4. 常接/前置关联命令
- **之前**：该分支的改动一定要已经合并到其他分支（比如 `main`），或者你确定不要了。
- **之后**：如果这个分支也推送到过远程，可能还想用 `git push origin --delete 分支名` 删除远程分支。

---

## 模块五：代码同步与拉取（远程 → 本地）

**这个模块解决什么问题？**
把 GitHub 上别人更新的代码，或者你自己在另一台电脑上推送的代码，拉取到当前电脑上，保持代码最新。

---

### 【同步拉取-下载并合并】命令 18：`git pull`
#### 1. 总体解释
从远程仓库下载最新内容，并立刻合并到你当前的本地分支里。相当于 **下载 + 合并** 一键完成。

#### 2. 命令逐词/逐部分拆解
- **`git pull`**：拉取。等于 `git fetch`（只下载不合并）+ `git merge`（合并）的快捷方式。

#### 3. 完整使用示例
```bash
git pull
```
如果已绑定上游分支，它会自动从对应的远程分支拉取。也可以指定：
```bash
git pull origin main
```

#### 4. 常接/前置关联命令
- **之前**：确保本地工作区是干净的（改动都提交了），否则可能产生冲突。
- **之后**：如果拉下来有冲突，需要手动解决冲突再提交。

---

### 【同步拉取-只下载不合并】命令 19：`git fetch`
#### 1. 总体解释
只从远程仓库下载最新的代码和分支信息，**但不合并到你的工作区**。相当于先看看远程有啥新变化，看完再决定怎么合并。

#### 2. 命令逐词/逐部分拆解
- **`git fetch`**：取来，只下载远程数据，不动你的本地文件。

#### 3. 完整使用示例
```bash
git fetch origin
```
之后可以用 `git log origin/main` 看看远程分支上的提交记录，再决定何时合并。

#### 4. 常接/前置关联命令
- **之后**：想合并的时候，用 `git merge origin/main`，或者直接用 `git pull`。

---

## 模块六：撤销与回退操作（本地/远程代码后悔药）

**这个模块解决什么问题？**
写错代码、提交错了、推送错了，别慌！这里有各种“后悔药”，帮你回到正确状态。

---

### 【撤销-丢弃工作区改动】命令 20：`git checkout -- 文件名` 或 `git restore 文件名`
#### 1. 总体解释
把一个文件**还没 add（加入暂存区）的修改全部撤销**，恢复到上次提交时的样子。相当于“这个文件我改坏了，还我原始版本”。

#### 2. 命令逐词/逐部分拆解
- **`git checkout -- 文件名`**：传统写法，`--` 后面跟文件名。
- **`git restore 文件名`**：新写法，意思更明确——恢复文件。

#### 3. 完整使用示例
```bash
# 传统写法
git checkout -- index.html

# 新写法（推荐）
git restore index.html
```

#### 4. 常接/前置关联命令
- **注意**：这个操作**不可逆**，改坏的部分将永久丢失！用之前一定想清楚。
- **之后**：文件恢复后，可以重新编辑。

---

### 【撤销-从暂存区移除】命令 21：`git reset HEAD 文件名` 或 `git restore --staged 文件名`
#### 1. 总体解释
把某个文件从“暂存区”（购物车）里拿出来，但**你在工作区的改动还保留着**。相当于你 `add` 错了，想重新选择。

#### 2. 命令逐词/逐部分拆解
- **传统写法**：`git reset HEAD 文件名`。`HEAD` 指当前最新的提交，意思是将暂存区重置为和最新提交一样。
- **新写法**：`git restore --staged 文件名`，`--staged` 表示操作暂存区。

#### 3. 完整使用示例
```bash
# 不小心把 debug.log 也 add 进去了，现在把它从暂存区拿走
git restore --staged debug.log
```
此时 `git status` 会显示 `debug.log` 变成“未暂存的修改”。

#### 4. 常接/前置关联命令
- **之后**：可以继续修改，然后重新 `git add`，或者用上一个命令丢弃改动。

---

### 【撤销-修改最近一次提交说明】命令 22：`git commit --amend -m "新的提交说明"`
#### 1. 总体解释
修改**最新一次提交**的说明文字（或者补充漏掉的文件），前提是这个提交**还没有推送到远程**。

#### 2. 命令逐词/逐部分拆解
- **`git commit --amend`**：修补最新的提交，而不是新建一个提交。
- **`-m "新的提交说明"`**：直接覆盖之前的说明消息。如果不加 `-m`，会打开文本编辑器让你改。

#### 3. 完整使用示例
```bash
git commit --amend -m "修复了支付接口超时问题并更新了文档"
```
如果在 `amend` 之前又 `add` 了新文件，这次会把新文件也并进上一个提交里。

#### 4. 常接/前置关联命令
- **重要警告**：如果这个提交已经 `push` 到 GitHub 了，不要用 `--amend`，否则会让别人协作出问题。如果已经推送了，只能用“回退版本并强推”的方法，但危险，新手慎用。

### 【回退-撤销本地提交（保留改动）】命令 23：`git reset --soft HEAD~1`
#### 1. 总体解释
撤销最近一次 `commit`，但**你改的代码全都回到暂存区**，像没事发生过一样。适合“刚提交完发现漏了个文件”的情况。

#### 2. 命令逐词/逐部分拆解
- **`git reset --soft`**：软重置，只移动 HEAD 指针，不动暂存区和工作区。
- **`HEAD~1`**：表示“相对于最近一次提交，往回退 1 步”。

#### 3. 完整使用示例
```bash
git reset --soft HEAD~1
```
执行后，最近一次提交消失，改动乖乖待在暂存区，你补充文件后可以重新 `commit`。

#### 4. 常接/前置关联命令
- **之前**：确保这次提交还没推送到远程。
- **之后**：改完补充文件后，重新 `git add`，然后 `git commit`。

---

### 【回退-撤销本地提交（彻底删除改动）】命令 24：`git reset --hard HEAD~1`
#### 1. 总体解释
撤销最近一次提交，并且连**代码改动一起永久删除**，回到前一个提交的干净状态。这是**最猛的后悔药**，用之前千万小心。

#### 2. 命令逐词/逐部分拆解
- **`git reset --hard`**：硬重置，移动 HEAD 指针，并且把暂存区和工作区都强制重置成目标版本的样子。
- **`HEAD~1`**：回退一步。

#### 3. 完整使用示例
```bash
git reset --hard HEAD~1
```
这之后，你最后一次提交的所有代码改动都没了。

#### 4. 常接/前置关联命令
- **死了都要记住**：操作不可逆！建议先备份或确认提交不需要了。
- **如果已推送**：一般不用这个，而要用 `git revert` 创建新提交来抵消。

---

### 【回退-已推送的提交安全撤销】命令 25：`git revert 提交ID`
#### 1. 总体解释
创建一个**新的提交**，这个提交的内容正好是把某次旧提交的改动全部“反向操作”，从而抵消掉那次错误。**这是对远程仓库最安全的撤销方式**，因为它保留了完整历史记录。

#### 2. 命令逐词/逐部分拆解
- **`git revert`**：反转，不是删除历史，而是新增一个抵消改动的新提交。
- **`提交ID`**：你想撤销的那次提交的哈希值，可以用 `git log` 查看。

#### 3. 完整使用示例
```bash
# 先查看日志，找到想撤销的提交ID，比如 abc1234
git log --oneline
# 撤销这次提交
git revert abc1234
```
之后会弹编辑器让你写新提交说明，保存退出即可。然后 `git push` 就能把“撤销”同步到远程。

#### 4. 常接/前置关联命令
- **之后**：像普通提交一样，用 `git push` 把这个“反向提交”推送到远程，远程的错误就被修复了。

---

### 【临时保存进度】命令 26：`git stash`
#### 1. 总体解释
当你改了一半代码，突然需要切换分支去修个急 bug，但又不想现在就提交。`git stash` 帮你把所有未提交的改动“暂存”到一个临时区域，工作区立刻变干净。等你回来再用 `git stash pop` 恢复。

#### 2. 命令逐词/逐部分拆解
- **`git stash`**：储藏，把改动推进一个临时存放区。

#### 3. 完整使用示例
```bash
# 暂存当前所有未提交的改动
git stash

# 切去修 bug，修完回来
# 恢复到最近一次 stash 的状态
git stash pop
```

#### 4. 常接/前置关联命令
- **之后**：马上切换分支去处理其他事。处理完后切回原分支，`git stash pop` 把进度放回来。
